# Первый рубеж

## Понятие бизнес-логики в программных системах. Уровень бизнес-логики в многоуровневой архитектуре программных систем.
- Часть ИС, отвечающая за реализацию предметной области
	- В многоуровневых ИС выделяется в отдельный уровень, изолированный от остальных интерфейсами
- Уровень БЛ находится между инфрой и сервисным уровнем
- Формирует основную вычислительную нагрузку
- Сложна в проектировании
- Обычно многопоточная
- Часто распределенная
- При разработке часто используется компонентный подход

## Основные концепции, используемые при разработке бизнес-логики. CDI, IoC, управление транзакциями, безопасность, распределённая обработка данных.

### Основные концепции
Бизнес-логика:
- Формирует основную вычислительную нагрузку на железо
- Сложна в проектировании
- Обычно многопоточная, часто распределенная
- При разработке часто используется компонентный подход

### CDI & IoC
- Универсальные архитектурные паттерны — используются не только в БЛ
- Удобны для "нарезки" бизнес логики на компоненты и их изоляции друг от друга
- Удобны для управления зависимостями
- Обычно реализуются внешним по отношению к компонентам БЛ фреймворком или платформой (Spring, JEE)

### Управление транзакциями
- Транзакции — набор операций, выполняемых как единое целое
- e.g. перевод денег между счетами
- В БЛ транзакции — более широкое понятие, чем транзакции в БД
- Разные технологические стеки предлагают разные реализации управления транзакциями

### Безопасность
- "Закрыть" под http basic auth обычно недостаточно
- Политики безопасности задают сквозные правила доступа к компонентам архитектуры приложения
- Это применимо не только для веба
- Вторая важная задача политик безопасности — отображение набора прав доступа на набор ролей конкретного пользователя

### Распределенная обработка
- Обеспечивает возможности горизонтального масштабирования БЛ
- В общем случае — очень сложная задача
- Реализация сильно зависит от выбранной платформы/фреймворка
- "Каноничные" варианты — кластеризация и создание пулов компонентов
- Более "модные" варианты — SOA и MSA

##  Моделирование бизнес-процессов. BPM и BPMN.

### Моделирование БП
- Набор формальных подходов к описанию БП
- БП разбивается на состояния и переходы между ними 
- Существует ПО для визуальной отрисовки схем БП

### BPM и BPMN
BPM — философия, концепт
BPMN — официальная стандартная нотация; используется во всем мире; Набро символов и графических представлений, которые вместе показывают весь поток процесса, его действия, события, шлюзы, соединения, etc

##  Спецификация BPMN 2.0. Принципы составления и основные элементы моделей бизнес-процессов.
BPMN — система условных обозначений для моделирования БП
Транслируется в XML
Специфицирована Object Management Group
Актуальная версия 2.0.2 опубликована в 2014
Моделирование осуществляется путем составления диаграмм, состоящих из стандартизированных элементов
BPMN — язык моделирования БП, который является промежуточным звеном между формализацией/визуализацией и воплощением БП
Представляет из себя описание графических элементов, используемых для построения схемы протекания БП
Такая схема нужна, чтобы, как минимум, выстроить в соответствии с ней БП, и понятно и однозначно регламентировать его для всех участников
Моделирование BPMN позволяет впоследствии провести автоматизацию БП в соответствии с имеющейся схемой

Преимущества:
- Простота трансляции диаграмм в исполняемые модели с помощью языка формального описания БП
- Описание элементов BPMN является понятным для большинства участников БПов и часто не требует никаких дополнительных разъяснений
	- С помощью простого графического выражения можно составить конкретные регламенты, которые будут исполняться сотрудниками
- Наряду с тем, что описание BPMN 2.0 позволяет добиться понимания сотрудниками того, как происходят БПы, данную нотацию поддерживает большинство современных инструментов бизнес-моделирования, что позволяет импортировать готовые схемы бизнес-процессов в BPM-системы

#### Примеры
`application.yml`

```yaml
spring:
  profiles:
    active: dev
```

`application-dev.yml`

```yaml
app:
  name: "MyApp (Development)"

  url: "jdbc:h2:mem:devdb"

  password: "dev_password"
```

`application-prod.yml`

```yaml
app:
  name: "MyApp (Production)"

  url: "jdbc:mysql://prod.server:3306/proddb"
  username: "prod_user"
  password: "prod_password"
```

Когда активен профиль `dev`, Spring загрузит настройки из файла `application-dev.yml`. Если активен профиль `prod`, он возьмет настройки из `application-prod.yml`.

#### Переключение профилей

##### Через `application.properties`
Если вы хотите задать профиль по умолчанию, просто добавьте его в файл `application.properties`:
```properties
spring.profiles.active=dev
```

##### Через командную строку
Активировать профиль можно при запуске приложения:
```bash
java -jar myapp.jar —spring.profiles.active=prod
```

##### Через переменные среды
Можно задать профиль с помощью переменной среды:
```bash
export SPRING_PROFILES_ACTIVE=prod
```

## Распределённые транзакции, спецификация XA. 

### Общий случай
Транзакция — группа последовательных операций, представляющая собой логическую единицу работы с данными
Виды транзакций:
- Последовательные (обычные)
- Параллельные
- Распределенные
ACID
- Atomicy
- Consistency
	- Атомарность
- Isolation
- Durability

### XA
XA — X/OpenXA (eXtendede Architecture) — спецификация распределенных транзакций
Является индустриальным стандартом в реализации менеджеров транзакций
Определяет принципы взаимодействия с транзакционными ресурсами в распределенных ресурсах

Архитектура XA:
![Архитектура XA](images/Pasted%20image%2020260613163300.png)
RM — управляет конкретными ресурсами (e.g. СУБД, файловым хранилищем, etc)
TM — координирует работу RM и принимает решение фиксировать/откатывать транзакцию
AP — описывает бизнес-логику, управляет составом транзакций и используемыми в ней RM
В XA специфицировано взаимодействие только между RM и TM. За AP отвечает программист

### Виды транзакций в XA:
1) Локальные
	1) Используется только одно хранилище, роль TM отводится AP
2) Глобальные/распределенные
	- Несколько хранилищ
	- Управлением транзакций занимается координатор
	- Менеджер транзакций делит одну глобальную транзакцию на несколько выполняющихся локально веток/бранчей
	- При успешной фиксации на каждой из веток осуществляет фиксацию всей глобальной транзакции
	- В противном случае осуществляет откат
	- Этапы двухфазного коммита:
		1) Этап подготовки
			- Координатор посылает сообщение участникам распределенной транзакции о подготовке к транзакции
			- Это сообщение содержит уникальный номер транзакции (TID)
			- Когда участники получают сообщение, происходит проверка смогут ли они заифксирововать транзакцию
			- После этого отвечают координатору
			- При этом транзакция исполняется, но не фиксируется
			- На диске сохраняется состояние
		2) Этап фиксации
			- Координатор получает все ответы $	o$ все зафиксировано ? Координатор посылает участникам сообщение о фиксации : Участники получают сообщение об откате транзакции
3) Вложенные
	- Транзакция запускается внутри существующей 
	- Изменения фиксируются только при фиксации "верхней" транзакции

## Реализация транзакций в приложениях на базе Jakarta EE и Spring.

### JEE
Java EE предоставляет JTA (Java Transaction API):
- API для управления распределенными транзакциями
- Специфицирован
- Входит в состав JEE
- Контейнер содержит менеджер транзакций, манипулирующий ресурсами
- Реализует стандарт X/OpenXA
![JTA транзакции](images/Pasted%20image%2020260614165811.png)

#### Особенности
- Поддержка реализуется на уровне сервера приложений (DataSource) и JDBC- драйвера (XAResources и тд)
- На несовместимых с XA JDBC-драйверах распределенные транзакции рабоать не будут
- Два высокоуровневых режима управления транзакциями в EJB:
	1) Императивное (программное)
		- BMT — реализует программист
		- Начинает, завершает и отменяет руками с помощью 
	2) Декларативное
		- CMT
		- Реализуется с помощью аннотаций `@TransactionAttribute`
		- За выполнение отвечает контейнер
		- Границы транзакции определяются границами методов и используемыми аннотациями
		- Что делать при вызовах методов друг из друга, так же определяется аннотациями

## Реализация управления транзакциями в Spring. 

### Аннотация `@Transactional`. 

### Декларативное и программное управления транзакциями.

#### Декларативное
Через аннотацию `@Transactional`
Метод вызывается через прокси-объект либо CGLIB Proxy либо JDK Dynamic Proxy 

##### Жизненный цикл:
```java
@Transactional
public void processOrder() {
    // 1. Прокси открывает транзакцию
    repository.save(order);
    
    // 2. Выполняется бизнес-логика
    updateInventory();
    
    // 3. Если исключение — rollback
    // 4. Если успешно — commit
}
```

##### Ключевые параметры
```java
@Transactional(
    propagation = Propagation.REQUIRED,      // Поведение при существующей транзакции
    isolation = Isolation.READ_COMMITTED,     // Уровень изоляции
    timeout = 30,                             // Таймаут в секундах
    readOnly = false,                         // Только для чтения
    rollbackFor = Exception.class,            // Какие исключения вызывают rollback
    noRollbackFor = IOException.class         // Какие НЕ вызывают rollback
)
```

#### Программное
С помощью TransactionTemplate, e.g.:
```java
class ManualTransactionIntegrationTest {
	@Autowired
	private PlatformTransactionManager ptm;
	private TransactionTemplate tt;
	
	@BeforeEach
	void setUp() {
		tt = new TransactionTemplate(ptm)
	}
	
	void givenPayment_WhenNotExpectingAnyResult_ThenShouldCommit() {
		tt.execute(new TransactionCallbackWithoutResult() {
				@Override
				protected void doInTransactionWithoutResult(TransactionStatus status) {
					Payment p = new Payment();
					p.setReferenceNumber("Ref-1");
					p.setState(Payment.State.SUCCESFUL);
					entityManager.persist(payment);
				}
			}
		)
		assertThat(entityManager.createQuery("select p from Payment p").getResultList().hasSize(1))
	}
}
```

#### Немного об основных концепциях
Самая фундаментальная вещь — SecurityContextHolder
В нем хранится информация о текущем контексте безопасности приложения
Контекст включает в себя подробную информацию о пользователе, работающим с приложением
SS использует объект `Authentication` (пользователя авторизованной сесии)

User — это просто объект
В большинстве случаев он может быть приведен к классу UserDetails. UserDetails можно представить как адаптер между БД пользователей и тем, что требуется SS внутри SecurityContextHolder

Для создания UserDetails используется функциональный интерфейс `UserDetailsService` с методом:
```java
UserDetails loadUserByUsername(String username) throws UsernameNotFoundException
```

#### Цепочные вызовы
![Цепочка фильтров Spring Security](images/Pasted%20image%2020260614150200.png)
Запрос, проходя по цепочке фильтров, отлавливается фильтром, нас троенный на аутентификацию запросов на определенный URL с определенным типом запроса
Данный фильтр создает объект типа Authentication и направляет его в AuthenticationManager
- ProviderManager является дефолтной реализацией AuthenticationManager
- Если необходимо, можно написать свою реализацию AuthentificationManager
AuthenticationManager возвращает авторизованный объект типа Authentification с вложенными в него дополнительными сведениями о пользователе
ProviderManager делает это с помощью заданного набора AuthenticationProvider'ов, каждый из которых имеет метод `supports()`, определяющий, применим ли данный провайдер к данной реализации Authentication
Authentication возвращает то же самое, что и AuthenticationManager, т.е. проводит авторизацию пользователя, устанавливает его роли и другую информацию
Как правило, для этого используется интерфейс UserDetailsService или его расширение UserDetailsManager, который возвращает объект типа UserDetails, содержащий информацию о пользователе
В случае неудачной авторизации пробрасывается исключение типа AuthentificationException, в случае удачной возвращается авторизованный объект типа Authentication, который сохраняется в SecurityContext

#### Ключевые объекты контекста Spring Security
- SecurityContextHolder
	- В нем содержится информация о текущем контексте безопасности приложения
	- Контекст включает в себя подробную информацию о пользователе работающем в настоящее время с приложением
- SecurityContext
	- Содержит объект Authentication и, в случае необходимости, информацию системы безопасности, связанную с запросом от пользователя
- Authentication
	- Представляет пользователя с точки зрения SS
- GrantedAuthority
	- Отражает разрешения выданные пользователю в масштабе всего приложения
	- Разрешения обычно называют ролями
	- e.g. `ROLE_ANONYMOUS`, `ROLE_USER`, `ROLE_ADMIN.`
- UserDetailsService
	- Этот сервис просто вытаскивает пользователя по полученным данным из хранилища (e.g. БД, памяти, properties-файла)

## Технология JAAS. Основные понятия, конфигурационные файлы и API. Использование на уровне бизнес-логики, в т.ч. совместно с Spring Security.

### JAAS
Стандарт API и реализация сервиса информационной безопасности
Входит в состав Java SE
Появилась в Java 1.3
С точки зрения сисадмина — набор конфигов
С точки зрения программиста — API для проверки ролей и полномочий
JAAS — это про определение того кто запрашивает доступ к ресерсу и про вынесение решения "а может ли он этот доступ вообще получить?"
![JAAS архитектура](images/Pasted%20image%2020260614142629.png)

#### Конфиги JAAS
\*.login.conf — определяет, какие логин-модули необходимо задействовать в программе
\*.policy — определяет, какими привилегиями наделены пользователи или программы

#### Объекты JAAS
Subject — субъект аутентификации, носитель или обладатель прав. Источник запроса на выполнение некоторого действия
Principal — представление субъекта с некоторой точки зрения
Credential — то, чем субъект подтверждает, что это он

#### Login Module
API, реализация которой осуществляет аутентификацию и авторизацию пользователей в приложении
Содержит 5 абстрактных методов:
- initialize
- login
- commit
- abort
- logout

## Системы единого входа, способы их интеграции с информационными системами.

### Кратко
SSO -- сервис, который обеспечивает сквозную модель аутентификации для нескольких приложений или подсистем
Позволяет пользователем переходить из одной подсистемы в другую без повторной аутентификации
Может быть интегрирован с различными провайдерами аутентификации (e.g. смарт-карты, биометрия, etc) 
> SSO -- более широкое понятие, чем Web SSO

### Возможные пути построения
1) С использованием центра распределения ключей
	- e.g. Kerberos
2) С использованием аппаратных решений 
	- e.g. смарт-карты
3) Всякие корпоративные SSOшки
	- e.g. NTLM
4) Всякие опен-сурц решения на базе SAML или OPENID

## Kerberos: принцип работы, реализуемый механизм аутентификации.
Kerberos -- сетевой протокол аутентификации
Основан на криптографии симметричных ключей
Требует наличия центра распределения ключей
Большинство современных ОС включают реализацию поддержки Kerberos

### Ключевые особенности
- Предполагается, что клиентский и серверный ноды доверяют KDC
- Пароль никогда не передается по сети
- Доверие между клиентом и сервером основано на том, что ключ, с помощью которого они дешифруют сообщения, известен только им и KDC
- Клиент и сервер взаимно доверяют друг другу
- Сессионный ключ имеет ограниченный срок действия $	o$ время на всех узлах должно быть синхронизировано

## Web SSO: отличия от SSO "общего назначения", принципы работы, используемые стандарты и протоколы.

### Web SSO
Более простая вещь, чем "полноценный SSO" (а-ля kerberos)
Может лучше соответствовать профилю веб-приложения, но для сложных систем его возможностей может быть недостаточно
Набор возможностей сильно зависит от выбранного протокола и реализации
Часто может использоваться вместе с тем же Kerberos

![Web SSO](images/Pasted%20image%2020260614184644.png)

### Стандарты 
Их много.

#### SAML
Security Assertion Markup Language -- протокол реализации SSO
Описывает формат обмена данными аутентификации и авторизации между поставщиком учетных записей и сервисом
Основан на XML
Последняя версия вышла в 2005г
В многодоменных системах часто заменяется на что-то более удобное, e.g. OpenID Connect

#### OpenID Connect
Открытый протокол аутентификации и авторизации
Де-факто для авторизации пользоваткля используется протокол OAuth 2.0
Активно поддерживается провайдерами аутентификации -- Google, Microsoft, итд
Есть поддержка "из коробки" в Spring Security

#### OAuth 2

##### Основные понятия
- Resource Owner
	- Пользователь, данные которого защищены SSO
- Client
	- Приложение, которое запрашивает доступ к данным пользователя
- Authorization Server
	- проверка личности + выдача токенов
- Resource Server
	- API, предоставляющий данные, к которым требуется доступ
